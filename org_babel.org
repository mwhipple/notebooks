#+title: org-babel

Org Babel is a literate programming environment using org mode in [[id:436EB0B7-811E-4828-BBF7-4126517DD7C6][emacs]].

As evidenced by this repository, I'm fairly keen on it.
I'm also toying with expanding this into a wider project called [[id:0F78767D-42FF-4027-B3A2-26CA1898F788][obi]].


I first started using it several years ago, but largely shelved it in favor of
lighter touch solutions, but more recently with the rise of using AI for coding
the model provided by org-babel seems particularly appropriate - resulting in options
like the babel files capturing what amount to live lab notebooks that were used to
drive the work of the agents.

Post processing typically only works with sync blocks, and once async is started it stays async.

** ~:var~

Set the named variable to the provided value.
This seems to be supported by synthesizing relevant assignment statements.

*** String Literal

#+begin_src emacs-lisp :var foo="bar"
  (format "foo=%s" foo)
#+end_src

#+RESULTS:
: foo=bar

*** Numeric Literal

#+begin_src emacs-lisp :var foo=1
  (+ foo 2)
#+end_src

#+RESULTS:
: 3

*** Expression

#+begin_src emacs-lisp :var foo=(buffer-name)
  foo
#+end_src

#+RESULTS:
: 20260615111334-org_babel.org

*** Named Block

This can also make use of richer content like tables and lists.

#+name: foo
A line of text

#+begin_src emacs-lisp :var foo=foo
  foo
#+end_src

#+RESULTS:
: A line of text

*** Evaluated Block

Also just throw in a demonstration of passing the values across languages.

#+name: foo-block
#+begin_src emacs-lisp
  (concat "biz" "baz")
#+end_src

For this case parentheses can optionally be included after the block name.

#+begin_src python :var foo=foo-block
  return foo
#+end_src

#+RESULTS:
: bizbaz

* Practices
** Packaging Up Behaviors
:PROPERTIES:
:ID:       DF79A72C-570A-49C8-A566-D494C59F732C
:END:

A major driver fro using org-babel is that it keeps the machinery
exposed and allows working with some sizeable chunks. As part of this
there will be a balance between approaches like copypasta and more
in-built functionality.

The rule of thumb is likely to keep things either very dumb and simple
(and rely on copypasta) or have them clearly defined as built in to
the engine. The undesriable middle ground is where functionality is
both not clearly defined as something that is provided, nor is it
clearly visible.
** Async Execution

One underlying goal would be to have everything async by default. This
seems fairly likely to invite splitting out [[id:0F78767D-42FF-4027-B3A2-26CA1898F788][obi]], but I'll see how far
this can get by default. Much of this should be able to be driven by async.

* Definition of Header Args

Most of the packages define the header args, but for the languages themsleves
the need/value is unclear (there's likely more info within the core org-babel code).

(org-babel-execute-src-block &optional ARG INFO PARAMS EXECUTOR-TYPE)

Execute the current source code block and return the result.
Insert the results of execution into the buffer.  Source code
execution and the collection and formatting of results can be
controlled through a variety of header arguments.

With prefix argument ARG, force re-execution even if an existing
result cached in the buffer would otherwise have been returned.

Optionally supply a value for INFO in the form returned by
‘org-babel-get-src-block-info’.

Optionally supply a value for PARAMS which will be merged with
the header arguments specified at the front of the source code
block.

EXECUTOR-TYPE is the type of the org element responsible for the
execution of the source block.  If not provided then informed
guess will be made.

#+end_example
