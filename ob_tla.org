* Current State

The library itself currently supports ~pluscal~ blocks...as that's
what I'll typically use.

The blocks should have a ~:file~ argument which is a valid TLA+ name,
matching the module content.

Fix the verification of this

If ~:tlc~ is added as a header argument it will instead be translated
and run.

When ~:tlc~ is used configuration will also be generated based on the
arguments.
- ~:specification~ must be provided
  improve the verification
- ~:invariant~ can be specified one or more times

I have ideas around providing a tidier interface, but for now the
above does the trick.

** Quick aside on interface

Before jumping too deeply into implementaiton I should think about how
I'd want the different interfaces to work given how tla+ tends to be
executed.

Out of the gate I'd gravitate towards two languages, one for tla which
will typically be used for verification, and one for pluscal which can
be either in translate or verify mode. The parameters that would
populate the configuration file can be passed as header args: they may
be verbose but it seems like the use of inputs like lists and tables
may provide a particularly nice way to manage that.

** Execution Composition

As discovered when working with [[id:12BC5AB3-D75B-43BC-A51E-C8CD80AAE829][tla+]] generally...there's a bit of a
gap in terms of CLI use where there's not a particularly great story
around convenient execution.

The jar can be invoked...but typically also requires a config file to
be created. Both pieces seem fairly easy, so the only lacking piece is
really a packaged interface. So far, I'm not sure such an interface is
worthwhile (unless it were integrated into TLA+ itself which seems
likely to have some friction), and so just handling those pieces
individually seems like the way to go. In many cases like this one the
config file contents will be fed programmatically, so trying to wrap
that up in some way to produce a fairly minimal format also seems
somewhat overkill. Something like mustache could probably be a good
fit to capture the formatting.

So anyway...the plan will be to produce a config file as needed and
then invoke the jar.

** Let's Get the Basics Working

The path to the config file needs to be provided to avoid issues, so
to start we'll provide a blank one.

The spec file needs to match the module definition, so the temp file
approach used by C babel won't cut it.

Instead will directly set [[id:C3FDF080-68AF-4F2C-ACB7-0F53F8307EAB][default-directory]] and then write the
appropriate files.

#+begin_src emacs-lisp
  (defun org-babel-execute:tla (body params)
    (let* ((default-directory (org-babel-temp-directory))
  	 (spec-file (cdr (assq :file params)))
  	 (cfg-file (format "%s.cfg" spec-file)))
      (with-temp-file spec-file (insert body))
      (with-temp-file cfg-file (insert ""))
      (org-babel-eval
       (format "%s tlc2.TLC -config %s %s"
  	     (org-babel-tla/cmd-base)
  	       cfg-file
  	       spec-file)
         "")))
#+end_src

** pluscal

Define the pluscal language which provides expansion but can also
be passed through tlc.

If the ~:tlc~ argument is passed the specification will be verified
and the output will be that produced by TLC - if it is not provided
the output will be the translated content.

#+begin_src emacs-lisp
  (defun org-babel-execute:pluscal (body params)
    (let* ((default-directory (org-babel-temp-directory))
  	 (spec-file (cdr (assq :file params))))
      (with-temp-file spec-file (insert body))
      (org-babel-eval
       (format "%s pcal.trans %s"
  	     (org-babel-tla/cmd-base)
  	     spec-file)
       "")
      (if (assq :tlc params)
  	(let ((cfg-file (format "%s.cfg" spec-file))
  	      (cfg-content (tla/render-config params)))
  	  (with-temp-file cfg-file (insert cfg-content))
  	  (tla/tlc spec-file cfg-file))
        (with-temp-buffer (insert-file-contents spec-file)
  			(buffer-string)))))
#+end_src

** tla/tlc

Execution of TLA will be split out so that it can be invoked by either
pluscal or tla (where the latter just becomes an adapter).

#+begin_src emacs-lisp
  (defun tla/tlc (spec-file cfg-file)
      "Inoke TLC for the provided spec-file, cfg-file, and arguments.
  This should be run in the same directory as those files (using default-directory)."
      (org-babel-eval
       (format "%s tlc2.TLC -config %s %s"
  	     (org-babel-tla/cmd-base)
  	     cfg-file
  	     spec-file)
       ""))
#+end_src

** Config File Generation

[[https://mustache.github.io][Mustache]] will be used for config file generation. The template should
be moved elsewhere and reused for things like the python interface,
but it will start life here and be subsequently extracted.

This looks like a good reference for the format:
https://docs.tlapl.us/using:tlc:config_file

The params are an alist with keyword keys, and the emacs mustache
library needs to be configured accordingly.

#+begin_src emacs-lisp
      (require 'mustache)
      (defconst tla/config-template "{{#specification}}SPECIFICATION {{specification}}{{/specification}}
{{#invariant}}INVARIANT {{invariant}}{{/invariant}}
    ")
      (defun tla/render-config (cfg)
        (let ((mustache-key-type 'keyword))
          (mustache-render tla/config-template cfg)))
#+end_src
